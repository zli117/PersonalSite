Chapter 1: Overview

April 19, 2026 · Zonglin Li

The key insight that ties everything together: modern Linux is a federation of daemons coordinating via well-defined IPC, sitting on top of a kernel that exposes its state through several named APIs. Understanding the seams — where one layer stops and the next begins — is more useful than memorizing any single layer.

Layer 0: The kernel’s boundaries#

The kernel exposes itself to userspace through a handful of interfaces, each with a different philosophy:

[[Syscalls]] — the classical interface. read(), write(), socket(), ioctl()([[ioctl]]). Direct function calls into ring 0.

Filesystems as APIs — /proc, /sys, /dev, and the newer /sys/fs/cgroup, /sys/fs/bpf. These aren’t storage; they’re views into kernel state. cat /proc/net/tcp isn’t reading a file, it’s asking the kernel to serialize its TCP socket table as text.

  • /proc — process and system state, dates from early Unix, mostly text.
  • /sys ([[sysfs]]) — device and driver state, one-value-per-file, newer and more structured.
  • /dev — device nodes, each one a handle for ioctl-style interaction with a driver.

Netlink — a socket-based protocol for structured communication with kernel subsystems. Used for networking (NETLINK_ROUTE, NETLINK_GENERIC), udev events (NETLINK_KOBJECT_UEVENT), audit, nftables, etc. This is how ip, iw, nft, and NetworkManager talk to the kernel. More robust than parsing /proc/net/*.

BPF — a safe in-kernel VM for running user-supplied programs attached to kernel hooks (networking, tracing, security). Growing hugely in importance.

Character/block devices — drivers register nodes in /dev, userspace opens them and issues read/write/ioctl/mmap.

The important thing: “the kernel” isn’t one blob you talk to — it’s many subsystems, each with its preferred interface. Networking is mostly netlink. Storage is mostly block devices + sysfs. GPUs are DRM character devices + ioctls. Input is evdev character devices.

Layer 1: Drivers and the hotplug story#

Drivers live in the kernel (most of them), but driver policy — which driver to load, what to name the device, what permissions to set, what daemon to notify — is userspace’s job. This is the [[udev]] story.

What happens when you plug in a USB WiFi dongle:

  1. USB subsystem in the kernel enumerates the device: vendor ID, product ID, class.
  2. Kernel matches against loaded/loadable driver modules via modaliases. For WiFi, it might load mt76x2u.ko.
  3. Driver initializes the hardware, registers a net_device with the networking stack, registers a wiphy with the nl80211 subsystem.
  4. Kernel emits a uevent over the NETLINK_KOBJECT_UEVENT socket: “new device, here’s its sysfs path and properties.”
  5. systemd-udevd (running in userspace) receives the uevent. It runs rules from /usr/lib/udev/rules.d/ and /etc/udev/rules.d/ — these decide the device name (e.g., wlp194s0 via the predictable naming scheme), set permissions, create symlinks in /dev, and emit a processed uevent on a different netlink multicast group.
  6. Other daemons subscribed to udev events react: NetworkManager sees the new network device and decides what to do with it based on its config. systemd sees it and activates any .device units that depend on it.

So: the kernel loads drivers; udev names devices and dispatches notifications; daemons react. There’s no single orchestrator — it’s event-driven pub/sub.

You can observe it live:

udevadm monitor --kernel --udev     # kernel uevents + processed udev events
udevadm info /sys/class/net/wlp194s0   # all attributes udev sees for a device

Layer 2: [[systemd]] — the init system and the service manager and a bunch else#

systemd is the elephant. It’s PID 1 after boot, but “init” is only a fraction of what it does. The relevant model:

Units are configuration objects describing things systemd manages. Types:

  • .service — a daemon (NetworkManager.service, sshd.service)
  • .socket — a listening socket that can launch its service on demand
  • .timer — a periodic trigger (replaces cron)
  • .device — a device known to udev (auto-generated)
  • .mount / .automount — filesystems
  • .target — a grouping/synchronization point (multi-user.target = “system is up for users”)
  • .path — watch a path, trigger on changes
  • .slice / .scope — cgroup resource-limit containers

Units have dependencies (Wants=, Requires=, After=, Before=) and ordering. systemd builds a DAG and parallelizes aggressively.

Key feature: socket activation. systemd can own the listening socket for a service and only start the service when a connection arrives. This is how “services start on demand” works — e.g., you can SSH to a machine and sshd starts at that moment, instead of running continuously.

Key feature: cgroup integration. Every service runs in its own cgroup (v2 on Fedora), which is how systemd tracks process membership reliably (not PIDs, which can be forged/reused) and enforces resource limits.

Beyond init, systemd has absorbed a lot of functionality:

  • systemd-udevd — device manager (was separate, merged ~2012)
  • systemd-logind — user session and seat management
  • systemd-journald — structured logging (replaces syslog for many uses)
  • systemd-resolved — DNS resolver and stub
  • systemd-networkd — network configuration (alternative to NetworkManager, server-oriented)
  • systemd-timesyncd — simple NTP client
  • systemd-homed — portable user accounts
  • systemd-boot — UEFI bootloader
  • systemd-nspawn — lightweight container runtime

Whether this consolidation is good or bad is a subject of much argument. Pragmatically: on Fedora, systemd owns most of the glue, and learning its model pays off.

Interaction with users: systemctl status/start/stop/enable/disable/reload, journalctl for logs, loginctl for sessions, resolvectl for DNS, networkctl for networkd state.

Layer 3: [[D-Bus]] — the userspace RPC bus#

D-Bus is how userspace daemons talk to each other and to applications. It’s not magic; it’s a message-passing bus with:

  • A broker daemon (dbus-broker on modern Fedora, formerly dbus-daemon) that routes messages.
  • Two buses per typical system: the system bus (one instance, root-privileged, for system services — NetworkManager, systemd, BlueZ, firewalld, UPower, logind) and the session bus (one per user session, for desktop apps).
  • Well-known names like org.freedesktop.NetworkManager that services claim, so clients know who to address.
  • Objects, interfaces, methods, properties, signals — roughly analogous to CORBA or COM, but simpler.

What this buys us: a standard way for decoupled components to communicate. Instead of NetworkManager needing a bespoke IPC protocol and every client implementing it, everyone speaks D-Bus and uses code-generated bindings.

Concrete example: when you run nmcli device status, nmcli opens a connection to the system bus, calls org.freedesktop.NetworkManager.GetDevices, formats the reply. No shared library, no config file parsing — just a D-Bus call.

You can poke the bus directly:

busctl list                             # everyone on the system bus
busctl tree org.freedesktop.NetworkManager    # its object hierarchy
busctl introspect org.freedesktop.NetworkManager /org/freedesktop/NetworkManager
busctl call org.freedesktop.NetworkManager /org/freedesktop/NetworkManager \
    org.freedesktop.NetworkManager GetDevices

systemd itself is heavily D-Bus-driven; systemctl is essentially a D-Bus client for PID 1.

Polkit sits on top of D-Bus: when an unprivileged client calls a privileged method (e.g., “change system time”, “connect to this WiFi”), the service asks polkit “is this user allowed?”, and polkit evaluates rules (possibly prompting for auth). This is how desktop apps trigger privileged operations without being setuid or running as root.

Layer 4: The network stack specifically#

Now let’s layer the specifics of networking.

Kernel owns:

  • net_device objects (each interface)
  • routing tables (up to 256 of them, with policy routing)
  • netfilter hooks (where nftables/iptables/bpf attach)
  • socket layer (TCP/UDP/SCTP, Unix sockets)
  • conntrack (stateful connection tracking)
  • cfg80211/nl80211 (WiFi configuration abstractions)
  • mac80211 (shared softMAC implementation many WiFi drivers use)

You interact with the kernel via netlink. Tools:

  • ip (from iproute2) — NETLINK_ROUTE, covers links, addresses, routes, rules, tunnels
  • iw — NETLINK_GENERIC via nl80211, covers WiFi
  • nft — NFNETLINK, covers nftables
  • ss — sockets (replaces netstat, uses netlink not /proc)
  • tc — traffic control/qdiscs
  • ethtool — driver-level NIC configuration (mostly ioctl, some netlink)

Userspace daemons coordinating network state:

  • [[NetworkManager]] — the big one on desktops. Owns connection profiles, drives wpa_supplicant for WiFi auth, runs DHCP client (internal or dhclient/dhcpcd), updates /etc/resolv.conf or talks to systemd-resolved, applies routes, manages VPN plugins (WireGuard, OpenVPN), sets firewall zones via firewalld’s D-Bus API. Exposed on D-Bus as org.freedesktop.NetworkManager.

  • wpa_supplicant — does 802.11 authentication (WPA2/WPA3/Enterprise). NetworkManager spawns and commands it via D-Bus. Not a daemon you usually talk to directly.

  • systemd-resolved — DNS stub resolver listening on 127.0.0.53. Applications query it (via nss-resolve or direct), it forwards to the real DNS servers, caches results, handles per-interface DNS, DNSSEC, DoT. NetworkManager pushes per-link DNS into it via D-Bus.

  • firewalld — as you now know, owns the nftables ruleset via zones. Exposed on D-Bus.

  • dhcpcd / dhclient / NetworkManager’s internal DHCP client — one of these runs per DHCP-enabled interface. NM’s internal one is default on Fedora.

A concrete trace: you click “Connect” on a WiFi network in GNOME.

  1. gnome-shell’s network indicator is a D-Bus client of NetworkManager. It calls ActivateConnection.
  2. NetworkManager receives the call. Looks up the saved connection profile (keyfile in /etc/NetworkManager/system-connections/).
  3. NM asks polkit whether the calling user is authorized. Polkit prompts if needed.
  4. NM prepares the interface via netlink (ip link set up equivalent).
  5. NM starts wpa_supplicant (if not running) and commands it via D-Bus to associate with the SSID using the stored PSK.
  6. wpa_supplicant does the 802.11 association and 4-way handshake with the AP. On success, it notifies NM.
  7. NM starts its DHCP client on the interface. DHCP client sends DISCOVER, gets OFFER, negotiates, ends up with an IP, gateway, DNS servers.
  8. NM configures the IP and route via netlink.
  9. NM pushes DNS servers to systemd-resolved via D-Bus.
  10. NM applies the connection’s firewall zone to the interface via firewalld’s D-Bus API.
  11. NM emits a StateChanged signal. Everyone subscribed (gnome-shell, your status bar, any other NM clients) sees the update and redraws.

Every step is observable. If something broke, you can look at journalctl -u NetworkManager, journalctl -u wpa_supplicant, journalctl -u systemd-resolved, busctl monitor org.freedesktop.NetworkManager, nft list ruleset.

Layer 5: The audio stack (sidebar, but structurally similar)#

Quick aside, because it parallels the network stack closely and matters for GUI:

Kernel: ALSA (/dev/snd/* devices, controls via sysfs/ioctls).

Userspace daemon: PipeWire (modern Fedora default). Single daemon handling audio and video streams (webcams, screen capture). Replaces PulseAudio (still presents its API for compat) and JACK (also presents its API). Routes via a session manager (WirePlumber) that applies per-device/per-app policy.

D-Bus interfaces for media keys, volume control, session management. This is how your keyboard’s volume key ends up changing volume in a specific app.

Layer 6: The graphical user interface#

The Linux GUI stack is notoriously complex because of historical accumulation (X11 is old, Wayland is newer, they coexist) and because “GUI” bundles several distinct concerns that are separately replaceable.

The six things a “GUI” actually does#

A graphical session needs to handle:

  1. Talk to the GPU — allocate framebuffers, submit rendering commands, schedule compositing.
  2. Set display modes — “run the monitor at 3840×2160@60Hz, sRGB, HDR off.”
  3. Read input devices — keyboards, mice, touchpads, touchscreens, tablets.
  4. Compose windows — take rendered frames from each app, stack them, scale, display.
  5. Provide a windowing protocol — how apps and the compositor communicate (here’s my window, here’s your input event, resize me).
  6. Render widgets — the buttons, text fields, menus that apps are built from.

Each is a separable layer. Understanding where one ends and the next begins is the key to the whole stack.

Kernel: [[DRM KMS]], evdev#

DRM (Direct Rendering Manager) is the kernel’s GPU subsystem. Each GPU shows up as /dev/dri/card0 (modesetting/privileged) and /dev/dri/renderD128 (rendering). Userspace opens these and sends [[ioctl]]s to allocate buffers, submit rendering command streams, and wait for completion. Memory management, scheduling, and security (isolating clients) all live here.

KMS (Kernel Mode Setting) is the subset of DRM that handles display configuration: enumerate connectors (HDMI, DisplayPort), probe EDID, set resolutions and refresh rates, manage CRTCs, handle page flips for tear-free updates. Before KMS, mode setting happened in userspace X drivers — hence the classic “X crashed and my monitor went black” era. Now it’s in-kernel and robust; the display keeps working even if the compositor crashes.

evdev is the input subsystem. Every keyboard, mouse, touchpad, touchscreen, joystick, tablet surfaces as /dev/input/event*. The kernel standardizes on a single event stream (type, code, value triples) regardless of connection (USB, Bluetooth, I²C, PS/2). evtest lets you watch the raw events.

ls /dev/dri/                   # card0, renderD128
ls /dev/input/                 # event0..eventN, plus by-id/ and by-path/ subdirs
sudo evtest /dev/input/event5  # watch events from one device

Access to these device nodes is privileged, which is where [[logind]] and seats re-enter: logind grants your session ACLs on DRI and input devices when you log in at a physical seat. When you switch to another VT, your session loses these; the new session gets them. This is also why remote SSH sessions can’t drive the display (no seat, no ACLs).

GPU userspace: Mesa and libraries#

GPU drivers in Linux split into two halves:

Kernel half (amdgpu, i915/xe, nouveau, nvidia proprietary) — memory management, command submission, scheduling, power management.

Userspace half — takes app-level APIs (OpenGL, Vulkan, OpenCL) and compiles them to hardware-specific command streams the kernel driver will submit. For open-source drivers, this lives in Mesa, a massive project providing userspace implementations of all the major graphics APIs. Proprietary NVIDIA ships its own userspace libraries as a blob.

libdrm is the thin syscall-wrapper library that everyone uses to talk to the DRM interface; compositors and Mesa both depend on it.

glxinfo | head                        # GL renderer and version
vulkaninfo --summary                  # Vulkan devices and drivers
lspci -k | grep -A 2 -i vga           # which kernel driver owns the GPU

Display server: X11 vs Wayland#

Here’s the central fork in the Linux GUI story.

X11 (the X Window System, ~1984 design) was the dominant Linux display server for three decades. Its model: one big server process (Xorg) owns the display, arbitrates between clients (apps), and implements a network-transparent protocol. Apps are clients that send drawing commands over a socket; the server draws.

The key design choice X11 made: the server does the drawing. This was right in 1984 (clients might be on remote mainframes; the server is where the pixels are) but wrong by the 2000s (clients want to render their own pixels with GPUs; the server is just a middleman). X11 accumulated extensions (Composite, Damage, XRender, DRI2, Present) that let clients render to offscreen buffers and hand pixmaps to a compositor — but the core model was a liability.

X11 ran most Linux desktops from 1990 to ~2020. It still runs many today, but new development is on:

Wayland (2008-) — a redesign. Key differences:

  • No separate display server process. The compositor is the Wayland server. One process does compositing and handles the protocol.
  • Clients render their own buffers (via EGL/Vulkan on GPU, or software on CPU), then hand fds to the compositor. The compositor composites.
  • Protocol is stricter about security. In X11, any client could read any other’s window content, inject keystrokes, or screenshot the whole screen. In Wayland, clients are isolated by default; system-level operations (screen capture, global hotkeys, input injection) go through explicit protocol extensions or portals.
  • No network transparency. X11 could forward windows over the network. Wayland doesn’t, natively. If you need that, use VNC/RDP/waypipe.

The dominant Linux compositors today:

Compositor Desktop Toolkit alignment
mutter GNOME GTK
KWin KDE Plasma Qt
sway sway (i3-like tiling) independent
hyprland Hyprland (dynamic tiling) independent
weston reference implementation —
gamescope Steam Deck, gaming sessions —

Most have both an X11 backend (for legacy) and a Wayland backend.

XWayland is the bridge. It’s an X11 server that runs as a Wayland client, so X11 apps keep working inside a Wayland session. You end up with (for example) Electron apps or old Java GUIs running under XWayland, while native GTK/Qt apps talk Wayland directly.

echo $XDG_SESSION_TYPE              # x11 or wayland
echo $WAYLAND_DISPLAY               # e.g., wayland-0
echo $DISPLAY                       # :0 under X, or XWayland display under Wayland
loginctl show-session $XDG_SESSION_ID -p Type -p Remote

Input: [[libinput]]#

Whether you run X or Wayland, libinput is now the standard userspace input library. It takes raw evdev events and turns them into high-level gestures: tap-to-click, two-finger scroll, palm rejection, pointer acceleration curves, tablet pressure. The compositor embeds libinput (Wayland) or X.Org’s xf86-input-libinput driver wraps it (X11). Same logic, different host.

Configuration used to be a nightmare (per-driver xorg.conf snippets). libinput is far simpler and ships sane defaults; desktop settings panels expose a handful of knobs (tap-to-click, natural scrolling, acceleration).

Session bring-up: display managers and login#

The path from “power on” to “desktop”:

  1. systemd brings up graphical.target.
  2. display-manager.service starts — GDM (GNOME), SDDM (KDE), LightDM (XFCE et al.), or greetd.
  3. DM opens a DRM/KMS display, reads input, shows a login screen.
  4. You pick a session type (GNOME on Wayland, Plasma on X11, sway, etc.) and authenticate via PAM.
  5. DM asks logind to create a session; logind assigns a seat and grants ACLs.
  6. DM spawns the session — typically gnome-session or the equivalent — which starts the compositor, a settings daemon, a window manager (if compositor and WM are separate), and the user’s autostart apps.
  7. The compositor claims the Wayland socket ($XDG_RUNTIME_DIR/wayland-0) or the X display.
  8. Apps launched from now on find the compositor via environment variables.

Each of these steps is observable: journalctl -u gdm, journalctl --user, ps auxf under your uid.

The [[Wayland]] protocol model#

Worth a concrete example because Wayland’s architecture shows up everywhere.

A Wayland compositor is a server that exposes a Unix socket. Clients (apps) connect and speak the Wayland wire protocol: object-oriented messages defining interfaces (wl_compositor, wl_surface, xdg_toplevel, wl_seat, wl_keyboard). Clients create surfaces, attach buffers (rendered via GPU fds), commit frames; compositor composites and responds with input events and configure messages (resize requests, window state).

The core protocol is deliberately minimal. Most functionality lives in protocol extensions — some stable, some compositor-specific. xdg-shell is the dominant extension for window management. xdg-desktop-portal (below) covers the stuff that was free-for-all in X11 (screenshots, file choosers, inhibit idle).

Introspection tools:

wayland-info               # like xdpyinfo; dumps compositor capabilities
weston-debug               # live protocol trace (if running weston)
WAYLAND_DEBUG=1 <cmd>      # dump protocol messages for a client

WAYLAND_DEBUG=1 is the GUI equivalent of strace for a single app — you see every protocol message, same pedagogical value.

Portals: the sandboxing bridge#

Wayland’s strictness means apps can’t just “take a screenshot” or “pick a file” or “prevent the screen from locking” — those operations need to be mediated. The pattern: xdg-desktop-portal, a D-Bus service running on the session bus.

Apps call portal methods (org.freedesktop.portal.Screenshot, .FileChooser, .Inhibit, .Notification). The portal shows a UI for confirmation (file picker dialog, “this app wants to record your screen — allow?”) and returns the result. The portal is compositor-specific under the hood (GNOME’s portal, KDE’s portal, etc., via different backends), but the D-Bus API is uniform.

This is also the path for [[Flatpak]] apps: sandboxed apps have only D-Bus access to the session bus, and portals are how they reach anything outside their sandbox. Native Wayland apps also use portals for screenshots etc., even without sandboxing.

busctl --user list | grep portal
# org.freedesktop.portal.Desktop
busctl --user tree org.freedesktop.portal.Desktop

Toolkits: how apps are actually built#

The toolkit is what app developers actually program against — it provides widgets, layouts, theming, event loops, accessibility hooks. The toolkit handles the Wayland/X11 protocol details so apps don’t have to.

Major toolkits:

  • GTK (version 3 and 4) — GNOME’s toolkit, used by GNOME apps, XFCE, Cinnamon, many independent apps (Firefox has a GTK backend). C-based with bindings for everything.
  • Qt — KDE’s toolkit, but also widely used outside KDE (Telegram, VLC, OBS). C++ with QML for declarative UI. Very comprehensive.
  • Electron — Chromium + Node.js bundled as an app. Each Electron app ships its own browser engine. Memory-hungry but pragmatic; VS Code, Slack, Discord, many others. Usually runs under XWayland still.
  • Tauri — lighter-weight alternative to Electron using the system’s webview.
  • Flutter — Google’s cross-platform toolkit; has Linux support. Growing slowly on Linux.
  • Dear ImGui, egui, Slint, Iced, Druid/Xilem — various immediate-mode or native Rust/C++ toolkits, common in games and niche apps.
  • Tk, FLTK, wxWidgets — older or niche toolkits, still in use.

Toolkit choice cascades into visual style: GTK apps follow Adwaita (GNOME’s HIG), Qt apps follow Breeze (KDE’s), Electron apps do their own thing. “Linux doesn’t have a consistent look” is largely because of this.

Toolkits also embed fonts (via fontconfig + FreeType + HarfBuzz — font discovery, rasterization, shaping respectively), rendering (Cairo for 2D vector, or direct GPU via GL/Vulkan), accessibility (AT-SPI over D-Bus, letting screen readers work), and input methods (IBus or Fcitx over D-Bus for CJK/Unicode input).

Desktop environments#

A desktop environment is a curated bundle of compositor + session manager + settings daemon + file manager + terminal + stock apps, with a consistent theme and HIG.

  • GNOME — opinionated, clean, uses mutter as the compositor, GTK. Dominant default on Fedora, Debian, Ubuntu.
  • KDE Plasma — highly configurable, uses KWin, Qt. More traditional desktop metaphor with lots of knobs.
  • XFCE, MATE, Cinnamon — lighter, more classical. Largely X11 still.
  • LXQt — lightweight Qt.
  • Budgie, Pantheon — smaller polished DEs.

Or you can skip the DE and assemble your own: a compositor (sway, hyprland, river) + a bar (waybar) + a launcher (wofi, rofi) + a notification daemon (mako, dunst) + a terminal + whatever else. This is the tiling-window-manager path — more assembly required, tighter to your preferences.

Accessibility, IM, theming — the secondary stack#

Three more D-Bus-based services worth knowing:

  • AT-SPI — the accessibility toolkit bus. Screen readers (Orca) and assistive tools subscribe to app events via AT-SPI.
  • IBus / Fcitx — input method frameworks. Apps forward keystrokes to the IM daemon via a protocol; the IM returns composed characters. Essential for Chinese, Japanese, Korean, Vietnamese, etc.
  • xsettings / gsettings / kconfig — desktop-wide settings storage. Themes, font sizes, icon themes. Apps read these at startup and on change notifications.

The whole picture for a GUI operation#

Trace: “you press Ctrl+Alt+T to open a terminal in GNOME on Wayland.”

  1. Keyboard generates USB HID report → kernel’s USB HID driver → evdev event on /dev/input/event5.
  2. mutter (GNOME’s compositor) is reading evdev via libinput. libinput recognizes the raw event and emits a cooked key event.
  3. mutter checks its keybinding table. Ctrl+Alt+T is bound to “launch default terminal” via GNOME’s settings.
  4. mutter calls into gnome-shell (same process, really — mutter is gnome-shell’s compositor component) to execute the action.
  5. gnome-shell uses its settings (org.gnome.desktop.default-applications.terminal, queried via gsettings/dconf over D-Bus) to find the binary — gnome-terminal.
  6. gnome-shell spawns gnome-terminal via g_spawn_async (ultimately fork+execve).
  7. gnome-terminal starts. It links against GTK, which connects to the Wayland socket ($WAYLAND_DISPLAY) and registers with the compositor.
  8. GTK creates a window: sends wl_compositor.create_surface, then xdg_surface.get_toplevel, requests decorations, commits an initial size.
  9. GTK renders the terminal widget into a buffer (via Cairo or GPU via GL).
  10. GTK sends wl_surface.attach(buffer_fd) + wl_surface.commit to the compositor.
  11. mutter receives the new buffer, composites it into the next frame, schedules a page flip via KMS.
  12. KMS tells the GPU to scan out the new framebuffer on the next vblank.
  13. Pixels hit the monitor.

Every step is observable: evdev events with evtest, libinput events with libinput debug-events, Wayland protocol with WAYLAND_DEBUG=1 gnome-terminal, GPU activity with amdgpu_top or intel_gpu_top, the gsettings read with dconf watch /, the spawn with strace -fe execve. All of it.

Common GUI troubleshooting#

  • Wayland session won’t start / falls back to X11 — journalctl -u gdm and journalctl --user. Usually a GPU driver issue (nvidia on older drivers, or missing Mesa support for your GPU).
  • Screen tearing or black screens — KMS is usually blameless. Check the compositor’s logs, GPU driver dmesg, maybe wrong DRM driver loaded.
  • App won’t run under Wayland — check if it’s XWayland (many still are). Try WAYLAND_DEBUG=1 <app> or GDK_BACKEND=x11 <app> to force XWayland.
  • Input device not recognized — sudo libinput debug-events sees everything libinput sees. If it’s not there, it’s an evdev/kernel issue.
  • Fractional scaling looks blurry — that’s XWayland apps; there’s no great solution yet for HiDPI on X11 clients.
  • Screenshot / screen record doesn’t work — ensure xdg-desktop-portal is running for your DE (systemctl --user status xdg-desktop-portal).
  • Copy/paste between X and Wayland apps breaks — clipboard bridges through XWayland, sometimes flakily. Compositor-specific.

Putting it all together#

The recurring pattern, now with GUI:

  • Kernel exposes hardware state and mechanism through syscalls, sysfs, netlink, DRM/KMS, evdev, BPF.
  • Low-level userspace (udev, libdrm, libinput, Mesa, wpa_supplicant) translates raw kernel events and wraps low-level APIs into usable shapes.
  • Coordinating daemons (systemd, NetworkManager, firewalld, resolved, PipeWire, logind, compositor) hold policy, configuration, session state, expose D-Bus APIs.
  • Toolkits and apps (GTK, Qt, Electron, plus every GUI app) call into those APIs. Talk to compositor via Wayland protocol, to system services via D-Bus, to kernel via syscalls.
  • Desktop environments are curated assemblies of the above with a consistent UI identity.

Four layers where you can intervene:

Want to… Touch this
Inspect kernel / hardware state /proc, /sys, ip, iw, nft, evtest, drm_info, lsusb, lspci
Change persistent config edit under /etc/ (system) or ~/.config/ (user), or use gsettings/dconf for GNOME
Make runtime changes with persistence use the daemon’s CLI (nmcli, firewall-cmd, systemctl, gsettings, wpctl)
Programmatically drive from another program D-Bus for most services; Wayland protocol for compositor; syscalls or netlink for kernel

Debugging tools across layers#

  • journalctl -xe — all logs; filter by unit, time, boot, priority.
  • journalctl --user — your session’s services and apps.
  • systemctl status / systemctl --user status — unit state.
  • busctl monitor <service> — watch D-Bus messages live.
  • WAYLAND_DEBUG=1 <app> — trace Wayland protocol messages for one client.
  • libinput debug-events — raw input events as libinput sees them.
  • evtest — deeper, raw evdev events per device.
  • wayland-info / xdpyinfo — compositor/display server capabilities.
  • amdgpu_top / intel_gpu_top / nvidia-smi — GPU activity.
  • drm_info — DRM device details, connectors, modes, planes.
  • glxinfo / vulkaninfo / eglinfo — graphics API capabilities.
  • strace -f -e execve,openat <cmd> — what syscalls.
  • bpftrace / perf — low-overhead kernel tracing.
  • nft monitor — watch nftables changes.
  • udevadm monitor — device events.
  • pw-top / wpctl status — PipeWire streams and devices.

Between these, almost any “why did that happen?” question about a running Linux desktop is answerable in a few minutes.

The minority report (revised)#

Things that look different on other stacks:

  • No systemd — Void (runit), Alpine (OpenRC), Gentoo (flexible). Kernel is the same; userspace glue differs.
  • No NetworkManager — servers often use systemd-networkd; embedded uses udhcpc + scripts.
  • No D-Bus on minimal systems — many daemons run without it; desktop integration suffers proportionally.
  • No Wayland — most servers and many embedded systems have no GUI at all, or X11 still, or direct DRM/KMS apps (Kodi, cage, gamescope).
  • No compositor at all — Steam Deck’s “gaming mode” uses gamescope as a minimal compositor; many embedded systems run a single DRM client directly.
  • Android — Linux kernel, completely different userspace. Its own display stack (SurfaceFlinger), init (init.rc), IPC (Binder, not D-Bus), app framework. Almost nothing above layer 1 ports over.
  • ChromeOS — Linux kernel, Chromium as the whole UI layer. Again, very different above the kernel.

The concepts port; the specific tools don’t. But mental models from one stack help understand the others — they’re all making similar choices about where to draw the privilege boundary, how to dispatch events, how to keep apps isolated.
